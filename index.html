Get Started

Quick Start

From clone to a chatting agent in six steps. You need Python 3.10+, a MongoDB instance (local or Atlas), and at least one LLM provider key.

Clone & install

Create a virtualenv and install the package with dev extras.

terminal
# Clone, create venv, install
git clone https://github.com/ashwin-gangadhar-mdb/mdb-agent-builder.git
cd mdb-agent-builder
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

Configure your environment

Copy the template and add your MongoDB URI, an LLM key, and a Flask secret.

.env
MONGODB_URI=mongodb://localhost:27017
MONGODB_DATABASE=agent_builder
ANTHROPIC_API_KEY=sk-ant-...
FLASK_SECRET_KEY=# python3 -c "import secrets; print(secrets.token_hex(32))"
MDB_ADMIN_TOKEN=# required only for admin endpoints

Scaffold config & run

make create-config writes a default config/agents.yaml; then start the dev server.

terminal
make create-config        # scaffold config/, logs/, prompts/
make validate-config      # sanity-check the YAML
make run                  # Flask dev server on :5000

curl http://localhost:5000/health
# → {"status": "healthy", "agent_loaded": true}
ℹ︎

Dev vs production. make run starts the single-process Flask dev server bound to 127.0.0.1. For production, use make serve-prod GUNICORN_WORKERS=8 or the Docker image — both run Gunicorn with MongoDB-backed state so sessions survive restarts and scale across workers.

Concepts

What's in the box

Every capability is declared in YAML and assembled at load time. Nothing here requires you to write a graph, a retry loop, or a policy check by hand.

◆

Five agent types

ReAct, Tool-Call, Reflection, Plan-Execute-Replan, and Long-Term Memory — pick one with a single agent_type key.

⇄

Multi-agent handoffs

Define several agents that transfer control to each other mid-conversation. Triage routes to specialists, specialists route back.

⊞

MongoDB-native tools

Vector search, full-text search, natural-language-to-MQL, a MongoDB toolkit, and any MCP server.

❋

Long-term memory

Episodic (verbatim recall) and observational (LLM-distilled facts) memory, both backed by Atlas Vector Search.

⛨

Governance & guardrails

Per-tenant access policies, prompt-injection detection, PII redaction, tool RBAC, and full audit logging.

⟳

Durable & scalable

MongoDB checkpointing and cross-worker session state. Run a fleet of Gunicorn workers against one consistent store.

Memory systems

Episodic

Stores verbatim conversation snippets — what was said and when — retrieved by semantic similarity.

Observational

An LLM distills raw chat into structured facts — preferences, goals, patterns — each stored as its own vector document.

Concepts

Governance, by configuration

Flip governance.enabled: true and the framework inserts a single policy-aware enforcement point into every agent — no per-tool wrappers, one auditable choke-point.

⛨

Access policies (RBAC)

Per-tenant, per-user, per-role policies resolved from MongoDB on each request, cached with a short TTL. Only permitted tools are callable.

⚠

Input guardrails

Blocked-topic rejection and prompt-injection pattern detection run before the agent ever sees the message.

⊘

PII redaction

Emails and phone numbers are redacted from both input and output. Message content never reaches the logs.

▤

Audit trail

Every allow/deny decision and chat completion is written to an agent_audit_events collection with tenant, user, and thread context.

⚠

Deployment note. The API has no built-in end-user auth — tenant_id/user_id are asserted by the client. Deploy behind a trusted gateway that authenticates callers and injects identity. Scope any nl_to_mql MongoDB user to read-only.

A complete agent, end to end

This is the full config for a governed enterprise-support agent: a local LLM, a vector-search tool over support docs, tenant-scoped retrieval, blocked topics, and audit logging. One file produces a running, policy-enforced service.

config/agents.yamlgoverned_enterprise_support
embeddings:
  - name: local_embeddings
    provider: huggingface
    model_name: sentence-transformers/all-MiniLM-L6-v2

llms:
  - name: local_llm
    provider: ollama
    model_name: llama3.1
    temperature: 0.1

tools:
  - name: support_docs
    tool_type: vector_search       # semantic search over Atlas
    connection_str: ${MONGODB_URI}
    namespace: mdb_examples.support_docs
    embedding_model: local_embeddings

governance:
  enabled: true
  connection_str: ${MONGODB_URI}
  default_policy:
    permissions: [tools.call.support_docs]   # tool RBAC
    blocked_topics: [payroll export]          # input guardrail
    retrieval_filters: { classification: [public, internal] }
  audit: { enabled: true }                 # every decision logged
  state: { enabled: true }                 # cross-worker sessions

agent:
  name: governed_enterprise_support
  agent_type: react
  llm: local_llm
  tools: [support_docs]
  system_prompt_path: ./prompts/governed_support_system.txt

Request lifecycle

What actually happens between a client's POST /chat and the response. Governance steps (3, 5, 8) are active only when governance.enabled: true.

Edge admission. Flask checks the 1 MB body cap, rate-limit bucket (keyed on IP + tenant + user), and the X-Requested-With CSRF header.
Identity binding. tenant_id, user_id, and roles from the request body become an IdentityContext.
Policy resolution. The MongoDB policy provider returns the matching AccessPolicy (cached with a ~5s TTL) for this tenant/user/role.
Input guardrail. Blocked topics and prompt-injection patterns are checked; PII is redacted. A block returns 403 before the LLM runs.
Agent invocation. The compiled LangGraph runs the LLM⇄tool loop. The policy is passed through the configurable dict so per-tenant rules apply without rebuilding the graph.
Policy-aware tool calls. Each tool call passes through one enforcement node — denied tools are blocked, retrieval is tenant-scoped, MQL/MCP tools require a tenant_filter.
Output guardrail + persistence. The response is PII-redacted, then history is written to the MongoDB state provider and the LangGraph checkpointer.
Audit + respond. Allow/deny decisions and agent.chat.completed events are logged; the JSON response returns to the client.

API endpoints

Deployment

Multi-worker (Gunicorn)

terminal
make serve-prod \
  GUNICORN_WORKERS=8 \
  PORT=5000

Enable state: or governance: in YAML so workers share conversation history through MongoDB.

Docker

terminal
make docker-build
make docker-run \
  PORT=5000 \
  GUNICORN_WORKERS=8

The image runs Gunicorn via startup.sh, mounts your config/, and exposes a /health healthcheck.
